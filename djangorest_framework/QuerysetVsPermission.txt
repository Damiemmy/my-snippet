# 🧠 Understanding DRF Queryset vs Permissions (Deep Dive)

## 🔥 The Situation

You received this error:

```json
{"detail": "No Product matches the given query."}
```

Instead of:

```json
{"detail": "You do not have permission to perform this action."}
```

---

# 🧠 Why This Happens

Your `get_queryset()` is:

```python
def get_queryset(self):
    if self.request.user.is_authenticated:
        return Product.objects.filter(user=self.request.user)
    return Product.objects.all()
```

---

## 🔍 What DRF Does Internally

When you send:

```http
PUT /products/1/
```

DRF runs:

```python
obj = self.get_object()
```

Which internally becomes:

```python
obj = Product.objects.filter(user=request.user).get(id=1)
```

---

## 💥 The Core Problem

If:

* Product `1` belongs to **User A**
* You are **User B**

Then:

```python
Product.objects.filter(user=UserB).get(id=1)
```

👉 ❌ No match found

So Django raises:

```json
"No Product matches the given query."
```

---

# ⚠️ Why You Didn’t See Permission Error

Because:

👉 The object was **never found**, so:

* `IsOwnerOrReadOnly` permission was **never executed**

---

# 🧠 DRF Request Lifecycle (VERY IMPORTANT)

| Stage | What Happens                      |
| ----- | --------------------------------- |
| 1     | Queryset tries to find the object |
| 2     | If found → permission check runs  |
| 3     | If NOT found → 404 error          |

👉 You failed at **Stage 1**, not Stage 2.

---

# 🔐 Is Your System Secure?

👉 YES — and even better than expected.

You now have **two layers of protection**:

---

## ✅ Layer 1: Queryset Filtering

```python
Product.objects.filter(user=self.request.user)
```

* Prevents access to other users’ data
* Hides existence of objects

---

## ✅ Layer 2: Object-Level Permission

```python
obj.user == request.user
```

* Ensures only owners can modify
* Acts as backup security

---

# 🔥 Why This Is Actually GOOD

Returning:

```json
"No Product matches the given query."
```

is sometimes **more secure** than:

```json
"You do not have permission"
```

Because:

* It hides whether the object exists
* Prevents attackers from guessing valid IDs

---

# ⚠️ When Will You See Permission Error?

You will ONLY see:

```json
"You do not have permission to perform this action."
```

IF your queryset allows access to the object.

---

## 🔧 Example:

```python
def get_queryset(self):
    return Product.objects.all()
```

Now:

* Object is found ✅
* Permission check runs ❌
* Permission error is returned

---

# 🧠 Final Understanding (LOCK THIS IN)

👉 You currently have:

* **Hidden objects (via queryset)** ✅
* **Protected objects (via permissions)** ✅

This is **enterprise-level backend design**.

---

# 🧭 Key Takeaways

* Queryset controls **visibility**
* Permissions control **access**
* DRF checks queryset **before** permissions
* If object isn’t found → permissions never run
