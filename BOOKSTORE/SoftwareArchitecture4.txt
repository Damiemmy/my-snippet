SOFTWARE ARCHITECTURE (THINK LIKE A SENIOR ENGINEER)

❓ What are we building exactly?
A system where:
    - Users can register and login
    - Users can browse books
    - Users can add books to cart
    - Users can place orders
    - Admin can manage books
    - Users can review books
    - Payments can be processed (later)
    - System sends emails in background

🧩 STEP 1 — CORE ENTITIES (THE REAL WORLD OBJECTS)

    Every system is just “things” + “relationships”.

    We identify the “things” first:

    👤 1. User: A person using the system.
        Two types:
            - Customer
            - Admin
    📚 2. Book: The main product.
    🏷️ 3. Category: Grouping of books.
        Example:
            - Tech
            - Fiction
            - Finance
    ✍️ 4. Author: Who wrote the book.
    🛒 5. Cart: Temporary storage before buying.
    📦 6. Order: Final purchase.
    ⭐ 7. Review: User feedback on books.
    💖 8. Wishlist: Saved books for later.
    💳 9. Payment: Money tracking (we simulate first, integrate later)


🧠 STEP 2 — RELATIONSHIPS (THIS IS WHERE MOST BEGINNERS FAIL)
Now we connect the dots.🧠 STEP 2 — RELATIONSHIPS (THIS IS WHERE MOST BEGINNERS FAIL)
Now we connect the dots.

    👤 USER → BOOK

        A user:does NOT own books
        but can:
            - review books
            - buy books
            - save books

        So:

            User → Reviews → Book
            User → Orders → Book
            User → Wishlist → Book


    📚 BOOK → AUTHOR

        One Author → Many Books

        So:

        Author (1) → (∞) Books

    🏷️ BOOK → CATEGORY
        One Category → Many Books
    🛒 CART

        Cart belongs to user:

        User (1) → (1) Cart
        Cart → CartItems → Books
    📦 ORDER
        User (1) → (∞) Orders
        Order → OrderItems → Books
    ⭐ REVIEW
        User → writes → Review → Boo

🧠 STEP 5 — BACKEND ARCHITECTURE DECISION

    We use:

    DRF ViewSets

    Because:
        - Faster development
        - Clean structure
        - RESTful automatically
        - Scales well
        
    We also use:Service Layer Pattern
    Instead of putting logic inside views:
        - views → calls services → handles business logic

    Example:
        order_service.py
        payment_service.py

🧠 STEP 6 — SECURITY DESIGN (VERY IMPORTANT)

We will enforce:

1. Authentication
JWT required for most endpoints
2. Authorization
Users can only access their own data

Example:
User A cannot see User B orders
3. Object-Level Permission
if obj.user != request.user:
    deny_access()
4. Queryset Filtering
Order.objects.filter(user=request.user)
